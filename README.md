# Loom
**Understanding, out loud.**

## The problem

I understand things better when I talk them through out loud than when I read a long explanation. Most AI tutors just hand you a wall of text and expect it to stick. Loom is built for people like me: you speak, Loom speaks back, and when it helps to see an idea rather than just hear it, Loom builds that too.

## What Loom does

Loom is a voice-first learning environment. You ask a question out loud, Loom answers out loud, and, when it decides a concept is better understood visually than described, it generates the right kind of comprehension artifact live: a manipulable simulator for quantitative relationships, a connection map for how ideas relate to each other, or a plot board for simple formulas and data. When no artifact would help yet, Loom stays on the board and simply adds a concept, a link, or a source.

As the explanation develops, Loom can also surface up to three short comprehension checks. They remain visible as a record of progress and are marked as resolved when the learner demonstrates the relevant understanding.

The principle behind all of it: Loom adapts the medium to the content, not the other way around.

## Running It Locally

### Prerequisites

- Node.js 20 or later
- npm

### Mock Mode: No API Key Required

Mock mode is the recommended first run. It is deterministic, requires no credentials, and remains the fallback for demo recording.

```bash
git clone <repository-url> loom
cd loom
npm install
cp .env.example .env
npm run dev
```

Open the URL printed by Vite, normally `http://localhost:5173`.

Keep `VITE_LOOM_API_MODE=mock` in `.env`. The supplied sessions cover projectile motion and a discursive connection-map conversation.

### Real API Mode

Real mode has been smoke-tested locally with a funded OpenAI account. It needs a local API key and two processes.

1. Create a project-root `.env` from `.env.example`.
2. Add `OPENAI_API_KEY=...` locally. Do not commit this file.
3. Set `VITE_LOOM_API_MODE=real`.
4. Start the local proxy in one terminal:

```bash
npm run dev:api
```

5. Start Vite in a second terminal:

```bash
npm run dev
```

The API path sends recorded audio to STT, receives a conversation reply and a controlled artifact plan, updates the board, then synthesizes Loom's reply with TTS. The configured default TTS voice is Cedar.

If the API is unavailable, malformed, or out of credit, Loom visibly switches to a stable mock response instead of leaving the board empty.

### Sample Sessions

- **Why does the ball arc?** develops from spoken orientation into a controlled projectile simulator.
- **Choice under pressure** develops into a free-will versus determinism connection map.
- The real path can also choose a controlled plot board for simple quantitative relationships, or remain on the motherboard when a visual artifact would not clarify the idea.

### Known Limitations

- Mock mode is deterministic by design and is the reliable recording fallback.
- Real mode requires local OpenAI billing and a browser microphone permission.
- The real limit is not which topics Loom can discuss. In real mode, Loom is not scripted to either demo example: it reasons about the nature of each concept. It draws through a controlled set of renderers, however, rather than an arbitrary canvas generated by the model. This is a deliberate safety choice: the model decides what to show and why, but never generates raw coordinates, SVG paths, or executable code; a local renderer always owns the drawing.
- Session history is local to the browser; there is no account system or cloud sync.

## Architecture

Loom is a Vite + React + Tailwind application. Its primary interface is a spoken learning scene with a visible transcript as support.

Artifacts are controlled rather than generated as executable UI. The model returns a validated JSON plan that configures existing local renderers:

- **ProjectileSimulator** receives bounded numeric inputs such as launch angle and velocity. The renderer computes the trajectory locally; the model never supplies coordinates or a pre-computed path.
- **ConnectionMap** receives semantic nodes, labeled links, title, and reading path. The renderer owns layout and drawing.
- **PlotBoard** receives one or two bounded data series, either as a linear, quadratic, or exponential formula or as explicit data points. Its configuration also includes axis labels and ranges, concise annotations, and up to two slider controls that target a declared coefficient. The renderer locally samples formulas into points, applies slider values to the relevant coefficient, maps data values to the chart coordinate system, and draws the SVG paths. The model never supplies SVG paths or screen coordinates.

The planner may also choose `stay-on-board`, so Loom can add only a concept, link, formula, source, or comprehension cue when an artifact would be premature.

Mock sessions are scripted for repeatability. In real mode, Loom is not tied to either example session: for each learner turn, the configured fast tier, `gpt-5.6-terra`, returns the spoken teaching reply plus board actions, comprehension checks, resolved check IDs, and title; the configured stronger tier, `gpt-5.6-sol`, decides whether the next idea should stay on the board, return to it, continue an active artifact, or show a supported live artifact. This separates conversational latency from structured planning depth.

The voice layer uses server-side STT and TTS endpoints. The transcript remains visible as support, while the board receives a compact set of semantic actions rather than full conversational text. Alongside each reply, Loom can receive zero to three short comprehension checks when enough context exists; resolved checks are retained and visibly marked rather than discarded. When a learner asks for current information, documentation, or links, the conversation request can use web search; verified result URLs are extracted into the board's clickable Sources section, while the TTS reply only says that sources were added. API failures and invalid structured output fall back to the stable mock path for the current turn.

## How I Collaborated With Codex And GPT-5.6

### Where Codex accelerated the work

Codex scaffolded the Vite/React/Tailwind project, implemented the `ProjectileSimulator`, `ConnectionMap`, and `PlotBoard` renderers, built the JSON schema and validation layer between the model and the renderers, wired the STT/TTS voice layer and the web-search-to-Sources pipeline, and iterated fast on the mock-to-real transition while keeping the mock path intact as a fallback.

### Where I made the key decisions

- Wrote a structured brief before any code, and required Codex to propose a phased roadmap and wait for confirmation rather than start building. I approved and constrained it.
- Set the build order myself: the projectile "mother scene" first, because it is the heart of the demo, the riskiest piece, and gives reusable UI scaffolding. Codex could not know that.
- Scope decisions: cut the PDF session export and cross-session concept linking; both belong to the full product vision, not a 7-day build.
- Product decision: voice-first with a supporting transcript, not text with optional voice.
- Content decision: chose projectile motion and free will vs. determinism deliberately for maximum contrast, to demonstrate medium-adapts-to-content.
- Engineering decision: mock-first, API later, so the experience could be validated independently and the demo has a fallback.
- Model routing decision: fast tier for dialogue, strong tier for structured planning.
- Scope expansion decision: after validating the two-artifact mother scene, I pushed to go beyond a fixed script, asking Codex to let the model reason about any concept and choose among the controlled renderers, including a new PlotBoard, rather than being told in advance which artifact to produce. I verified this with topics outside the two demo examples; for example, the French Revolution correctly stayed on the board with no forced artifact.
- Where I corrected Codex: it misread the GPT-5.6 tiers as agent roles rather than model tiers. I caught it and corrected the configuration; the tier names were then verified against official documentation.
- Design and brand: mine, including the identity, palette, and interface decisions.

### How GPT-5.6 contributed to the final result

GPT-5.6 drives the spoken dialogue in real time, reasons about the nature of each concept to decide which comprehension artifact, if any, fits it, produces the validated structured plan that configures the renderer, and, when the learner asks for current information or sources, performs web search and returns verified, clickable results.
